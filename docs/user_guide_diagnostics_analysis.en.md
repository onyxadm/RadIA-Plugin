# User Guide: Diagnostics & Code Analysis

This guide details how to use **Rad IA**'s advanced tools to locate and debug compilation errors, analyze exception stack traces, and perform static quality and security analyses on your Object Pascal code.

---

## 1. Compilation Debugger (Smart Build Debugger)

Often, errors flagged by the Delphi Compiler (DCC) are cryptic or hard to debug at first glance (such as syntax errors resulting from incorrect type declarations or string literals violating the 255-character limit). Rad IA integrates with the IDE's message window to help.

### How to Use:
1. Compile your project in the IDE (`Ctrl + F9` or `Shift + F9`).
2. If compiling fails, navigate to the **Messages** window (at the bottom of the Delphi IDE).
3. **Right-click** on the compilation error line.
4. In the pop-up menu, select **Rad IA: Explain and Fix Error**.
5. Rad IA will open the chat sidebar, automatically sending the error code, the compiler message, and the source code snippet of the affected line.
6. The AI will explain the root cause of the compilation failure and suggest the corrected code, ready to be applied.

---

## 2. Stack Trace Assistant

Debugging errors that occur in staging or production environments based purely on textual exception logs is exhausting. Rad IA has a dedicated context-aware assistant to decode this data.

### How to Use:
1. In the IDE, open the `.pas` file (unit) where the stack trace indicates the error occurred.
2. In the Rad IA chat input, type `/stacktrace` followed by the error log. Example:
   > `/stacktrace EAccessViolation in TInvoiceService.ProcessInvoice at line 122`
   *(You can also paste the full stack trace generated by tools like **MadExcept** or **EurekaLog**).*
3. Rad IA will capture the code of the active unit currently open in the editor, send it to the AI along with the provided call stack, and request a correlation.
4. The AI will analyze the code corresponding to the lines pointed out in the log and return a technical assessment indicating where the failure (null pointer, array index out of bounds, etc.) occurred and how to fix it.

---

## 3. Memory Leak & Anti-pattern Scanner

Memory management in Delphi for desktop platforms is manual. The absence of `try..finally` protection blocks is the main cause of memory leaks. Rad IA allows you to audit your code statically to prevent these issues from reaching production.

### How to Use:
1. Open the Pascal file you want to audit in the editor.
2. To analyze the entire unit: type `/bugs` in the chat sidebar (or use the keyboard shortcut `Ctrl + Shift + B`).
3. To analyze only a specific block: select the code snippet, right-click, and choose **Rad IA -> Find Bugs**.
4. The AI will perform a static scan focusing on:
   * **Memory Leaks**: Local instantiation of classes (such as `TList<T>`, `TStringList`, connection objects) that lack a matching `try..finally` block guaranteeing their release (`.Free`).
   * **Exception Handling**: Empty or overly generic `try..except` blocks that silence critical system errors.
   * **SOLID & Clean Code Principles**: Highly coupled code, methods with excessive lines of code, or classes with multiple responsibilities.
5. Refactoring suggestions will be displayed in the chat panel, allowing you to use the *Smart Diff* comparison tool to apply changes directly.

> [!WARNING]
> **Limitations of Static Memory Analysis:**
> The memory leak validation performed by the AI during the `/bugs` command is a **static, structural analysis of the code**. It scans for suspicious syntactic patterns (like the absence of `try..finally` blocks). This **does not replace** Delphi's dynamic runtime execution test mechanisms. It is highly recommended to keep `ReportMemoryLeaksOnShutdown := True` enabled in debug mode within your application, or use profiling tools (such as FastMM4, MadExcept, or EurekaLog) for runtime validation.
